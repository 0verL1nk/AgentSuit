# AgentSuit 模块化/插件化工程最佳实践

## 1. 文档目的

本文档聚焦 AgentSuit 这类“模块化 + 插件化 + 多 runtime / 多 base agent 适配”的系统，给出一套可持续开发的软件工程实践。

目标不是讨论某一个 adapter 怎么写，而是先把整体工程纪律定下来，避免项目在后续接入更多 base agent、plugin、memory provider、tool provider 后出现以下问题：

- 扩展点越来越多，但没有稳定契约
- core 被 runtime 细节反向污染
- adapter / plugin 之间互相深度依赖
- 版本兼容关系模糊，只能靠人工试错
- 测试只能测 happy path，升级容易炸
- 发布包来源不透明，后续难做供应链治理

本文档适用于：

- `packages/core`
- `packages/schema`
- `packages/runtime`
- `packages/adapter-*`
- `packages/plugin-*`
- 未来的 `memory provider`、`tool provider`、`policy provider`

---

## 2. 总体结论

对于 AgentSuit，这类系统要长期可维护，核心不是“插件越灵活越好”，而是：

1. 先定义稳定公共契约，再开放扩展点。
2. 所有扩展都通过显式接口接入，禁止依赖内部实现细节。
3. 宿主负责生命周期、状态、鉴权、隔离和报告，插件只负责自己那一层能力。
4. 兼容性必须是机器可判定的，不能靠 README 口头说明。
5. 发布、测试、观测、弃用策略必须从一开始纳入设计，否则后期成本会指数上升。

## 3. 设计原则

### 3.1 Public Contract First

公共 API 要先于实现稳定下来。语义化版本规范明确要求，使用 SemVer 的软件必须先声明 public API；发布后的版本内容不能被原地修改，破坏兼容性的改动必须走 major 版本。

对 AgentSuit，这意味着：

- `@agentsuit/schema`、`@agentsuit/core-contracts`、`@agentsuit/plugin-api`、`@agentsuit/adapter-api` 必须先成为明确的公共契约包
- 插件和 adapter 只能依赖这些公共契约，不允许直接 import 宿主内部文件
- 改公共接口时，必须同步更新版本、兼容矩阵和迁移说明

### 3.2 Extension Point First, Not Internal Hook First

Backstage 的经验很值得借鉴：扩展应该通过显式 extension point 暴露，而不是让外部模块“知道内部结构后自己插进去”。

对 AgentSuit，这意味着：

- core 对外暴露一组明确 extension points
- 例如：`BaseAgentAdapter`、`ExposurePlugin`、`MemoryProvider`、`ToolProvider`、`PolicyProvider`
- extension point 的输入、输出、错误模型、生命周期都要显式定义
- 任何“偷用内部 utils / internal types / internal state”的做法都视为架构违规

### 3.3 Host Owns Lifecycle

PF4J 的实践很清楚：插件的加载、启停、启用、禁用、卸载都应该由宿主统一控制，不能交给插件自由漂移。

对 AgentSuit，这意味着：

- 只有 runtime host 可以决定 plugin load / start / stop / disable
- plugin 不能自己偷偷开后台线程或额外监听端口而不注册
- 所有插件都要接受统一健康检查和启动失败回滚
- adapter / plugin 的状态要进入统一报告体系

### 3.4 Keep Cross-Cutting Concerns Split

OpenTelemetry 对可扩展系统的一个关键经验是：跨切关注点必须拆成 API、SDK/Runtime、Contrib 三层，否则会把整个系统拖进耦合泥潭。

对 AgentSuit，可以直接映射为：

- `API / Contracts`
  - schema
  - plugin api
  - adapter api
  - event model
- `Runtime / Host`
  - session manager
  - lifecycle
  - state store
  - auth / routing
- `Contrib`
  - plugins
  - adapters
  - optional providers

不要让 contrib 反向定义 core 的行为。

## 4. 推荐的架构边界

### 4.1 推荐依赖方向

```text
schema/contracts
    ^
    |
core ---------> runtime
 ^               ^
 |               |
adapter-*        plugin-* / provider-*
```

约束如下：

- `schema` 只定义数据模型和校验规则，不依赖 runtime
- `core` 负责 planning、capability classification、report model
- `runtime` 负责会话、生命周期、状态、插件宿主
- `adapter-*` 只能依赖 contracts、schema、core，不能依赖某个 plugin 的内部实现
- `plugin-*` 只能依赖 plugin api 和 runtime 提供的公共上下文
- `plugin-*` 之间禁止直接 runtime 依赖

### 4.2 严禁 Deep Import

Node 官方文档对 package `exports` 的建议很适合这里：公共包入口要显式收口，禁止消费者随意 import 未导出的内部路径。

对 AgentSuit 的建议：

- 所有可被外部依赖的包都启用 `exports`
- 只暴露明确子路径，例如：
  - `@agentsuit/plugin-api`
  - `@agentsuit/adapter-api`
  - `@agentsuit/schema`
- 不允许：
  - `@agentsuit/runtime/dist/internal/foo`
  - `@agentsuit/core/src/bar`
- 若新增 `exports` 会影响既有导入路径，应视为 breaking change

### 4.3 插件不是“任意代码入口”

插件机制应是“受约束的扩展”，不是“用户可以随便把任意代码塞进来”。

建议：

- 先开放有限的 extension point，再考虑动态插件发现
- v0.x 阶段优先使用静态注册或受控注册表
- 等 `plugin-api` 稳定后，再开放第三方动态装载

这比一开始就做重量级动态插件系统更稳。

## 5. 统一 Capability 模型

AgentSuit 后续一定会遇到能力不对齐问题，尤其是：

- prompt 注入方式不同
- tool / MCP 接入方式不同
- skills / workflow 支持程度不同
- memory 有的 runtime 没有原生支持
- coworker / subagent 语义不一致
- approval / policy 行为差异很大

因此建议系统级采用统一 capability 状态：

- `native`
  - base agent 或目标 runtime 原生支持
- `degraded`
  - 可以有损映射
- `plugin-backed`
  - 宿主通过 AgentSuit 的 plugin/provider 自行补齐
- `blocked`
  - 无法安全实现，必须显式阻断

### 5.1 Memory 应被视为 Provider，而不是固定模块

这点尤其重要。

不要假设所有 base agent 都有 memory。更合理的设计是：

- `memory contract`
  - 定义 memory 的语义
- `memory provider`
  - 定义谁来实现 memory

示例：

- `native-memory-provider`
  - 使用 base agent 原生 memory
- `local-file-memory-provider`
  - 由 AgentSuit 本地文件实现
- `redis-memory-provider`
  - 由 Redis 持久化实现
- `rag-memory-provider`
  - 由外部向量检索或知识库实现

这样 memory 就天然是可拔插能力，而不是绑死在 base agent 里。

## 6. 包与版本治理

### 6.1 明确哪些包是稳定公共包

建议尽早区分：

- `stable public packages`
  - `@agentsuit/schema`
  - `@agentsuit/plugin-api`
  - `@agentsuit/adapter-api`
- `internal packages`
  - `@agentsuit/runtime-internal`
  - `@agentsuit/core-internal`

如果不做这一步，后面外部 plugin 会自然依赖到错误的包。

### 6.2 用 SemVer 管公共契约

建议：

- 只有公共契约包严格承诺 SemVer
- internal 包可以更快迭代，但不能被第三方直接依赖
- `0.x` 阶段允许接口快速迭代，但仍要把 breaking change 写清楚
- 一旦开放第三方 plugin 开发，就应该尽快把核心契约推进到 `1.0`

### 6.3 对插件宿主用 `peerDependencies`

npm 官方文档明确给了这条路：对于需要和宿主交互但不应被插件自己安装一份的依赖，用 `peerDependencies`；对于可选依赖，可用 `peerDependenciesMeta.optional` 标为可选。

对 AgentSuit 的建议：

- plugin 对 `@agentsuit/plugin-api` 使用 `peerDependencies`
- adapter 对 `@agentsuit/adapter-api` 使用 `peerDependencies`
- 某些只在特定环境启用的宿主能力，可以标记为 optional peer dependency

这样可以避免：

- 每个 plugin 带一份自己的 host API 副本
- 运行时出现类型兼容但实例不兼容的问题

### 6.4 `optionalDependencies` 只用于真正可缺省能力

npm 文档也明确说了，`optionalDependencies` 安装失败不应该让整体安装失败，但程序自己必须能处理依赖缺失。

所以在 AgentSuit 中：

- `chat` 插件依赖某个 UI 包，如果缺失还能降级，可以用 optional
- 但核心 `plugin-api`、`schema` 这类契约包不能用 optional

### 6.5 `overrides` 只在根项目使用

npm 官方文档说明 `overrides` 只在 root `package.json` 生效，已发布依赖内部的 `overrides` 不参与最终解析。

因此建议：

- 安全修复或统一版本时，用 root `overrides`
- 不要指望某个 plugin 自己通过 `overrides` 控制整个宿主依赖树

## 7. 插件生命周期与隔离

每个 plugin / adapter 都应该有明确生命周期：

1. `discover`
2. `load`
3. `validate`
4. `start`
5. `healthy`
6. `degraded`
7. `stop`
8. `unload`

建议：

- 加载前校验 manifest、版本要求、能力声明
- 启动失败时支持回滚，不让系统进入“半启动”状态
- 端口、路由、background task 必须注册
- 插件必须能被禁用
- 插件状态变更必须进入事件流和报告

PF4J 还特别强调，宿主应维护“插件要求的最小系统版本”。这对 AgentSuit 很重要：

- 每个 plugin / adapter 都应声明 `requires`
- 宿主在加载前就拒绝不兼容版本
- 不要等运行时方法签名炸了才发现不兼容

## 8. 配置与 Schema 最佳实践

### 8.1 配置必须有 Schema

Backstage 的实践是：插件如果要暴露配置项，应该同步定义配置 schema，而不是只在代码里偷偷读取。

对 AgentSuit 建议：

- `service`、`exposures`、`memory`、`tool`、`policy` 都应有 schema
- 每个 plugin 的 config 都应提供 schema 与默认值
- 配置错误要在启动前失败，而不是运行中才报错

### 8.2 区分静态配置与动态状态

不要把动态 session 数据、事件偏移量、插件运行时缓存写进静态配置。

建议拆开：

- `config`
  - 可提交、可审查
- `state`
  - 不可提交、运行态生成
- `reports`
  - 可审计、可复盘

### 8.3 配置覆盖优先级要固定

建议固定优先级：

`CLI flags > project config > suit manifest defaults > package defaults`

并写进文档，不允许不同模块自行发明优先级。

## 9. 测试策略

### 9.1 不只测实现，要测 Contract

最先要补的不是更多 e2e，而是 contract tests：

- `plugin-api` contract tests
- `adapter-api` contract tests
- capability mapping tests
- report schema tests

只测实现很容易导致“内部看起来都能跑，换一个 plugin 就炸”。

### 9.2 统一测试宿主 Harness

Backstage 的测试实践很值得借鉴：它提供统一 test backend harness 和 mock services，让插件在接近真实宿主的环境里测试。

对 AgentSuit 的建议：

- 提供 `startTestRuntime()` 这类统一测试宿主
- 允许挂假 adapter、假 plugin、假 memory provider
- 让插件测试跑在近似真实生命周期下

### 9.3 建立兼容矩阵测试

至少维护这些矩阵：

- host version × plugin-api version
- runtime version × adapter version
- plugin × transport mode
- memory provider × persistence backend

不是所有组合都要全量测，但关键组合必须自动化。

### 9.4 测远程交互时 mock 网络，而不是 mock 自家 client

Backstage 官方测试文档建议，和外部服务交互时尽量 mock HTTP 边界，而不是只 mock 本地 client。

对 AgentSuit 意味着：

- 测 web/chat/A2A 插件时，优先从 HTTP / WS / SSE 边界测
- 测第三方服务接入时，mock 网络响应而不是绕过 transport 层

### 9.5 Docker 烟测要成为发布门槛

因为你的首发目标是 `docker run` 即可用，所以至少要有：

- 单容器启动烟测
- 健康检查烟测
- 基础 session 往返烟测
- 至少一个插件启动烟测

## 10. 可观测性与报告

插件化系统后期最容易死在“出了问题不知道在哪一层”。

因此建议从一开始统一事件和观测：

- 生命周期事件
- session 事件
- tool 调用事件
- plugin 状态事件
- adapter 映射事件
- degrade / blocked 事件

OpenTelemetry 的一个关键启发是：API、SDK、Contrib 要分层，语义约定也要单独维护。

对 AgentSuit 的建议：

- 维护统一事件模型
- 为常见字段建立稳定语义约定
  - `instance.id`
  - `session.id`
  - `plugin.name`
  - `adapter.name`
  - `capability.kind`
  - `capability.status`
- 日志、指标、报告尽量复用同一套字段命名

不要每个 plugin 自己定义一套事件名。

## 11. 发布、供应链与来源治理

SLSA 对 provenance 的定义很适合插件生态：产物应可追溯到“从哪里来、何时构建、如何构建”。

对 AgentSuit 的建议：

- 每个发布包至少保留：
  - source revision
  - build timestamp
  - package version
  - dependency lock
- 中期目标：
  - 生成 provenance
  - 对发布物做签名或校验摘要
  - 让宿主在加载第三方 plugin 时校验来源

首阶段即使做不到完整签名，也至少要做到：

- deterministic package metadata
- checksum
- 可审计构建来源

## 12. 团队开发流程建议

### 12.1 任何新增扩展点都要配 ADR

新增下列内容时，必须有 ADR 或设计记录：

- 新 extension point
- plugin-api breaking change
- adapter capability 模型变化
- event model breaking change
- 新增动态装载机制

### 12.2 变更分层评审

建议 code review 时分三层看：

1. 契约是否变化
2. runtime / plugin 边界是否被破坏
3. 测试矩阵是否补足

### 12.3 弃用策略

建议明确：

- `deprecated` 至少保留一个 minor 周期
- 删除前提供迁移说明
- CLI / runtime 在弃用期给 warning

## 13. 对 AgentSuit 的具体落地建议

### 13.1 尽快补四个公共包

建议优先显式建立：

- `@agentsuit/schema`
- `@agentsuit/plugin-api`
- `@agentsuit/adapter-api`
- `@agentsuit/runtime-test-kit`

### 13.2 先做受控插件生态，再做开放生态

建议阶段化：

- `v0.x`
  - 官方维护插件与 adapter
  - 静态注册或受控发现
- `v1.x`
  - 开放第三方插件开发
  - 引入签名、来源校验、兼容矩阵

### 13.3 Memory 先做 provider 模型

不要把 memory 当成某个 runtime 私有特性。

建议直接定义：

- `MemoryCapability`
- `MemoryProvider`
- `MemoryStore`

这样以后不管 Claude Code 是否原生支持 memory，都可以走：

- native
- plugin-backed
- blocked

三种路线。

### 13.4 把“支持矩阵”变成仓库事实

建议维护机器可读兼容文件，例如：

```text
docs/compatibility/
  base-agent-matrix.yaml
  plugin-api-matrix.yaml
  capability-status-matrix.yaml
```

不要只在文档里写“理论支持”。

---

## 14. 推荐落地顺序

1. 固化 `schema / plugin-api / adapter-api` 三类公共契约。
2. 给所有对外包加 `exports`，禁止 deep import。
3. 定义 plugin / adapter 生命周期与 `requires` 版本字段。
4. 建立 `startTestRuntime()` 测试宿主。
5. 建立 capability 状态机：`native / degraded / plugin-backed / blocked`。
6. 把 memory、tool、policy 都按 provider/plugin 思路建模。
7. 引入兼容矩阵与 Docker 烟测。
8. 再考虑开放第三方动态插件加载。
